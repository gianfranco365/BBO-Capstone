# Datasheet for the BBO Capstone Dataset

*Following the Datasheets for Datasets framework (Gebru et al., 2021)*

---

## Motivation

**For what purpose was the dataset created?**  
To support a structured Bayesian black-box optimisation (BBO) capstone project within the Imperial College London Executive Master in Machine Learning & Artificial Intelligence. The dataset provides initial observations for eight synthetic unknown functions, serving as the starting point for iterative GP-based optimisation across sequential submission rounds.

**Who created the dataset and on whose behalf?**  
Created and curated by Imperial College London as part of the BBO Capstone project portal. Accessed and used by Gian Franco Cattaneo (COO, SALOV S.p.A.) as course participant.

**Any other comments?**  
The dataset is intentionally opaque — the analytical form of each function is unknown to the participant. This mirrors real-world black-box scenarios (proprietary algorithms, complex simulations, biological processes).

---

## Composition

**What do the instances represent?**  
Each instance is an input–output pair: a d-dimensional vector `x ∈ [0, 1)^d` (the query point) and a scalar `y = f(x)` (the function output). All eight functions are treated as maximisation problems.

**How many instances are there in total?**

| Function | d | Initial observations (N₀) | Submitted queries (R1–R5) | Total (est.) |
|----------|---|--------------------------|--------------------------|-------------|
| f1 | 2 | 10 | 5 | 15 |
| f2 | 2 | 10 | 5 | 15 |
| f3 | 3 | 15 | 5 | 20 |
| f4 | 4 | 30 | 5 | 35 |
| f5 | 4 | 20 | 5 | 25 |
| f6 | 5 | 20 | 5 | 25 |
| f7 | 6 | 30 | 5 | 35 |
| f8 | 8 | 40 | 5 | 45 |

*Initial N₀ values are estimates from session history; exact counts confirmed by portal .npy files.*

**Does the dataset contain all possible instances or is it a sample?**  
It is a sparse sample of a continuous input space. The true function domain is [0, 1)^d for each function. Even at N=45 (f8), the space is vastly undersampled at 8 dimensions.

**Does the dataset contain data that might be considered confidential?**  
No. All data is synthetic, generated by the portal for academic purposes.

**Is any information missing from individual instances?**  
No missing values. All submitted queries return a single scalar output from the portal without exceptions.

**Are there any known errors, sources of noise, or redundancies?**  
- **Observation noise**: present at low levels in some functions (WhiteKernel noise level fits 1e-4 to 0.1 across functions).  
- **Near-duplicate queries**: R1 and R3 for f7 and f8 are near-identical points (submitted independently), confirming output stability rather than introducing noise.  
- **Boundary artefacts**: f4 exhibits catastrophic output collapse (−27 to −30) at boundary region (x1→1, x2/x3→0), which may reflect a domain-specific feature of the synthetic function design rather than noise.

## Collection Process

**How was the data collected?**  
Two sources:

1. **Initial data**: provided as `.npy` files by the Imperial College BBO portal at project start. Loaded via `np.load()`.
2. **Submitted query data**: generated by the participant using GP-EI / UCB acquisition functions, submitted one point per function per round to the portal, and returned as input–output pairs plus cumulative `.npy` files.

**What mechanisms or procedures were used to collect the data?**  
- Surrogate model: Gaussian Process with Matérn-5/2 ARD kernel + WhiteKernel
- Acquisition function: Expected Improvement (maximisation convention) with multi-start L-BFGS-B optimisation (35 restarts, ξ per-function)
- Round 4 used minimisation EI (error, corrected in Round 5)

**Who was involved in the data collection process?**  
Queries designed by Gian Franco Cattaneo using scikit-learn, scipy, and numpy. Portal evaluation performed by Imperial College systems.

**Over what timeframe was the data collected?**  
March–April 2026 (Rounds 1–5, Weeks 12–16 of the Executive Master programme).

## Preprocessing / Cleaning / Labelling

**Was any preprocessing of the data done?**  
- `StandardScaler` (zero mean, unit variance) applied to input arrays `X` before GP fitting in all rounds
- `normalize_y=True` applied within `GaussianProcessRegressor`
- No clipping, imputation, or transformation of output values `Y` (used raw as returned by portal)

**Is the software used to preprocess the data available?**  
Yes — see `requirements.txt`. All preprocessing occurs inline within the notebooks.

## Uses

**What tasks has the dataset been used for?**  
- Gaussian Process surrogate model fitting (regression)
- Bayesian optimisation via Expected Improvement and UCB acquisition
- Neural network surrogate training (MLPRegressor 2×32 ReLU) for gradient/sensitivity analysis
- Binary classification framing ("good" vs "bad" outputs via median threshold)
- SVM margin distance analysis for support-vector-like point identification

**Is there anything about the composition of the dataset that might affect future uses?**  
- With N≤45 per function, GP posteriors in high dimensions (d=6, d=8) carry significant uncertainty. Any analysis claiming definitive landscape characterisation from this dataset should be treated with caution.
- The dataset is specific to these eight synthetic functions and does not generalise to other optimisation problems.

**Are there tasks for which the dataset should not be used?**  
- Production deployment decisions (the functions are synthetic and do not represent real industrial processes without domain validation)
- Training general-purpose ML models (dataset is too small and domain-specific)

## Limitations

1. **Small N**: 4–5 submitted observations per function (plus initial data). GP confidence intervals are wide; ARD length-scale estimates are unstable at N<10 per dimension.
2. **Single-point queries**: one query per function per round imposes slow convergence relative to batch BO approaches.
3. **No gradient information**: the black-box constraint prohibits analytical gradient access. NN finite-difference gradients are an approximation with bias at sample boundaries.
4. **Function-specific scope**: insights (e.g., f4 boundary collapse, f7 sparse structure) are specific to these synthetic functions and do not constitute general principles.
5. **Maximisation assumption**: if any function has not been negated by the portal (i.e., a minimisation function submitted as-is), the acquisition function direction would be incorrect. This is assumed to be correctly handled by the portal per project specification.
